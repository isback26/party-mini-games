## 3. 369 규칙 상태 정리

### easy
- 기존 369 규칙
- 숫자 안에 3/6/9가 들어가면 박수
- 아니면 숫자 입력

### normal
- easy 규칙 포함
- 10, 20, 30 ... 처럼 10의 배수에서는 `만세`

### hard
- normal 규칙 포함
- 10의 배수는 `만세`
- 숫자 안에 3/6/9가 들어가면 그 개수만큼 박수
- 그 외에도 **3의 배수는 숫자 대신 박수 1회**

예:
- `3` → 박수 1회
- `33` → 박수 2회
- `369` → 박수 3회
- `10` → 만세
- hard에서 `12` → 박수 1회

## 4. 현재 369 입력/UI 상태 정리

### 입력 방식
- 숫자 입력은 자동 제출 구조
  - 1~9는 1자리 입력 즉시 제출
  - 10 이상은 필요한 자리 수가 차면 자동 제출
- 박수 입력은 연속 탭 기반
- 만세 입력은 전용 버튼 1회 입력
- 정답 타입/횟수를 UI가 직접 알려주지 않음

### 표시 방식
- 상단 우측은 `정답`이 아니라 **직전 플레이어의 실제 입력값** 표시
- `waiting_start` 중에는 상단 우측에 카운트다운 표시
- 카운트다운 종료 후 직전 입력 표시로 전환

### 버튼 배치
현재 369 입력 패널은 아래 순서로 정리되어 있다.
1. 숫자 입력창
2. 숫자패드
3. 박수 입력 / 만세 입력
4. 뒤로(백스페이스) / 전체삭제

이 배치는 스마트폰 실사용 기준으로 손가락 이동을 줄이기 위한 방향이다.

## 5. 오답/종료 흐름 정리
현재는 오답 입력 시 단순 종료가 아니라 아래처럼 동작한다.

- 실제 잘못 입력한 값이 서버 metadata에 남는다
- 상단 우측 표시도 마지막 실제 입력값 기준으로 유지된다
- 종료 메시지에서 정답과 오답을 함께 알려준다
  - 예: `정답은 "👏👏"인데 틀렸습니다.`
- 게임 종료 대화상자에 **다음 게임은 누구부터 시작인지** 표시된다

시간 초과 패배도 동일한 “다음 시작 플레이어” 계산 흐름에 반영된다.

## 6. 시작 플레이어 규칙 정리
- **첫 게임**은 방장이 시작 버튼을 누르고, 첫 턴도 기본 순서 시작
- **다음 게임부터**는 직전 게임에서 틀리거나 시간 초과로 패배한 사람이 첫 턴이 된다
- 현재 단계에서는 **시작 버튼 권한은 여전히 방장**에게 있다
- 즉, 바뀐 것은 “누가 시작 버튼을 누르느냐”가 아니라 “첫 턴이 누구냐”이다

## 15. 2026-03-30 눈치게임 MVP 1단계 진행 현황 업데이트

이번 작업에서는 369를 크게 흔들지 않고, 눈치게임 MVP 1단계의 핵심 구조를 우선 연결하는 방향으로 진행했다.

### 15-1. 이번 채팅에서 확인/반영된 내용

#### 서버/규칙 방향
- 눈치게임은 369/번데기와 달리 **턴제 방식이 아님**을 규칙으로 확정했다.
- 카운트다운 이후에는 **살아있는 플레이어 누구나 현재 숫자를 먼저 누를 수 있는 구조**로 정리했다.
- 같은 숫자를 **0.5초 이내에 여러 명이 누르면 모두 실패**하는 방향으로 서버 authoritative 판정 구조를 설계/연결했다.
- 잘못된 숫자를 누른 플레이어도 실패 처리 대상이라는 규칙을 명확히 반영했다.
- 인원 수가 N명일 때, 실제 성공 숫자는 `1 ~ (N-1)`까지이며 마지막 1명은 구조적으로 탈락하게 되는 규칙을 기준으로 잡았다.
- 실패가 1명이든 여러 명이든 발생하면 **그 판은 즉시 종료 후 새 판으로 다시 시작**하는 구조로 이해를 맞췄다.

#### 클라이언트/화면 방향
- 눈치게임도 숫자를 잘못 누를 수 있으므로, 입력 UI는 **단일 큰 버튼**이 아니라 **369와 유사한 숫자패드 재사용 방향**이 맞다고 판단했다.
- 다만 눈치게임에서는 박수, 만세, back, 전체삭제 같은 보조 입력은 불필요하므로 제거 방향으로 조정했다.
- 눈치게임 화면에 다음 정보 영역을 우선 추가했다.
  - 현재 숫자 / 목표 숫자
  - 직전 숫자
  - 성공 숫자 기록
  - 참가자 상태 보드(생존/탈출/탈락)
  - 방금 입력한 사람 + 숫자 중앙 강조 패널

### 15-2. 이번 채팅까지 실제 확인된 상태

#### 완료/부분 완료
- 눈치게임 전용 화면 진입 연결 완료
- 눈치게임에서 **방금 입력한 사람 + 숫자**를 크게 보여주는 강조 패널 추가 완료
- 참가자 보드 공통 컴포넌트 `_ParticipantSeatBoard` 도입 완료
- 369에도 공통 참가자 보드 적용 시작
- 직전 입력자 강조를 위한 UI 뼈대 확보

#### 아직 남아 있는 문제
- **모바일 세로 overflow 문제**가 눈치게임/369 양쪽 모두 완전히 해결되지 않았다.
- 특히 숫자패드 영역과 상단 정보 패널이 한 화면 안에서 길어질 때,
  스마트폰 세로 높이에서 `BOTTOM OVERFLOWED`가 다시 발생한다.
- 따라서 다음 단계 최우선 과제는 기능 추가가 아니라,
  **공통 게임 화면 레이아웃의 모바일 높이 최적화**이다.

### 15-3. 현재 판단
눈치게임 MVP는 아직 “규칙/입력/표시 구조를 연결하는 단계”이며,
핵심 서버 authoritative 방향성과 UI 정보 구조는 맞게 잡혔다.

하지만 모바일 실사용 기준으로는 아직 다음 항목이 완료되어야 한다.
1. overflow 없는 안정적인 세로 레이아웃
2. 정보 패널/참가자 보드/입력 패널 높이 재배치
3. 이후 원형/사각형 좌석 배치 등 연출형 UI 개선

즉, 지금은 눈치게임을 더 크게 확장하기보다
**모바일 화면 맞춤 안정화**를 먼저 끝내는 것이 맞다.

### 15-4. 다음 채팅 최우선 목표
다음 채팅에서는 아래 순서로 진행한다.

1. 최신 `app/flutter_app/lib/main.dart` 기준으로 현재 레이아웃 다시 확인
2. 눈치게임/369 공통 화면에서 발생하는 overflow 원인 분리
3. 입력 패널 높이, 상단 정보 패널, 참가자 보드 영역을 재배치
4. 스마트폰 기준 overflow가 완전히 사라질 때까지 작은 diff로 조정
5. 그 다음에야 원형 좌석 배치 등 고도화 UI로 넘어간다

### 15-5. 한 줄 결론
