## 3. 이번 채팅 기준으로 확인된 369 규칙 상태

### easy
- 기존 369 규칙
- 숫자 안에 3/6/9가 들어가면 박수
- 아니면 숫자 입력

### normal
- easy 규칙 포함
- 10, 20, 30 ... 처럼 10의 배수에서는 `만세`

### hard
- normal 규칙 포함
- 10의 배수는 `만세`
- 숫자 안에 3/6/9가 들어가면 그 개수만큼 박수
- 그 외에도 **3의 배수는 숫자 대신 박수 1회**

예:
- `3` → 박수 1회
- `33` → 박수 2회
- `369` → 박수 3회
- `10` → 만세
- hard에서 `12` → 박수 1회

## 4. 이번 채팅 기준 369 입력/UI 상태 정리

이번 창 동안 아래 방향이 정리되었다.

### 현재까지 도달한 방향
- 입력 화면은 스마트폰 기준으로 **숫자패드 + 빠른 입력 버튼** 구조를 유지한다.
- `만세 입력` 버튼이 추가되는 방향으로 정리했다.
- 로비에는 **369 전용 난이도 선택 UI**를 붙이는 방향으로 정리했다.
- 서버는 난이도 값을 받아 실제 게임 시작 시 반영하는 구조를 사용한다.

### 이번 창에서 중요하게 확인한 UX 이슈
- `숫자 제출` 버튼은 스마트폰에서 불편하다.
- 박수 1/2/3회 선택을 별도 선택창으로 고르는 것도 실전에서는 불편할 수 있다.
- 실제 게임에서는 **즉시 입력형에 가까운 구조**가 더 자연스럽다.
- 상단 우측에는 정답 표시보다 **직전 플레이어가 실제로 입력한 값**을 보여주는 방향이 더 맞다.

## 5. 이번 채팅 기준 서버/타이머 상태 정리

## 8. 다음 채팅에서 확정된 우선 방향

다음 채팅에서는 아래를 **한 번에 다 하지 않고 작은 단계로** 진행한다.

### 1순위
상단 우측 정답 표시 제거 → **직전 플레이어 입력 표시**로 바꾸는 설계를 먼저 시작한다.

이 단계에서 먼저 확인할 것:
- 현재 `gameState.metadata.expectedDisplayText`에 의존하는 표시 구조를 어떻게 바꿀지
- 서버 메타데이터에 `lastSubmittedText` 같은 값을 둘지
- waiting_start 카운트다운 표시와 충돌하지 않는지

### 2순위
`숫자 제출` 버튼 제거를 위한 입력 방식 재설계

목표 방향:
- 1~9는 숫자 누르면 즉시 제출
- 10 이상은 2자리 누적 후 자동 제출
- 턴 시간 안에 2번째 자리를 못 누르면 서버 timeout으로 종료
- normal / hard에서 십단위 만세 규칙과 충돌하지 않게 설계

### 3순위
박수 입력을 연속 탭 기반으로 재설계

목표 방향:
- 박수 1회가 정답이면 버튼 1번으로 즉시 제출
- 박수 2회 필요 시 2번 입력이 완료될 때 제출
- 박수 3회도 동일
- 시간 초과는 서버가 판정

### 4순위
`999` 도달 시 강제 종료 룰 추가

목표 방향:
- `999`까지 성공하면 더 이상 진행하지 않음
- `참 잘 했어요!` 메시지와 함께 게임 종료

## 9. 다음 채팅에서 사용할 구체 요구사항 정리

다음 창 목표는 아래 사용자 요구를 기준으로 한다.

1. 게임화면 오른쪽에는 정답 대신 **직전 사람이 누른 숫자/입력 결과**를 표시한다.
2. `숫자 제출` 버튼은 제거한다.
3. 1~9는 숫자 누르면 바로 입력, 10 이상은 필요한 자리 수가 모이면 자동 입력되는 방향으로 간다.
4. normal / hard의 만세는 한번 누르는 입력으로 유지하되, 숫자 입력과 규칙 충돌이 없게 한다.
5. 박수도 횟수 선택창이 아니라 **연속 입력형**으로 간다.
6. 2회/3회 박수가 필요한 경우, 시간 안에 그 횟수만큼 눌러야 한다.
7. `999` 성공 시 “참 잘 했어요!”와 함께 종료한다.
8. 다음 창에서도 작은 단계부터 진행한다.
