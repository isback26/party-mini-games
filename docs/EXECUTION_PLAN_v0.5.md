# EXECUTION_PLAN_v0.5.md

## 0. 이번 문서의 기준
- 이 문서는 **2026-04-12 현재 대화에서 확인된 진행 상태**를 반영한다.
- 실제 플레이 대상은 **스마트폰**이다.
- Windows는 **개발/테스트용**이다.
- 시간 판정은 반드시 **서버 authoritative** 구조로 유지한다.
- 다음 창에서도 **최신 첨부 파일 기준으로만** 판단한다.
- 확인되지 않은 파일은 추측하지 않고 요청한다.
- 수정안은 반드시 **unified diff 형식**으로 제안한다.
- 초보자 기준으로 설명한다.
- 한 번에 많이 수정하지 않고 **작은 단계**로 진행한다.

## 3. 리액션 시스템 현재 상태

### 3-1. 완료된 내용
- `ReactionBar` UI 추가 완료
- 좌석 보드(`ParticipantSeatBoard`) 위에 리액션이 보이도록 1차 구조 반영 완료
- 리액션은 **플레이어 카드 위에 overlay 형태**로 표시되도록 반영 완료
- 리액션은 **1.4초 후 자동 제거**되도록 반영 완료
- 같은 플레이어가 여러 번 누르면 동시에 여러 개 보일 수 있도록 반영 완료
- 서버를 통해 **같은 방 모든 참가자에게 리액션 동기화** 확인 완료
- 서로 다른 폰에서 리액션 표시 확인 완료

### 3-2. 현재 동작 구조
1. 사용자가 `ReactionBar` 버튼을 누른다
2. 클라이언트가 `reaction:send` 이벤트를 서버로 전송한다
3. 서버가 같은 방 참가자인지 확인한다
4. 서버가 `reaction:show`를 방 전체에 broadcast 한다
5. 각 클라이언트는 해당 `socketId` 플레이어 좌석 위에 리액션을 띄운다
6. 일정 시간이 지나면 자동으로 사라진다

### 3-3. 현재 리액션 UI 상태
- 로컬 미리보기 SnackBar 구조는 제거됨
- 좌석 카드 위에 리액션이 직접 보이는 구조로 전환됨
- 리액션 버튼은 텍스트보다 **이모지 중심 표시** 방향이 더 적합하다고 판단됨
- 369 입력 버튼에서 `박수`, `만세`는 텍스트보다 `👏`, `🙌` 중심으로 다듬는 방향이 적합함

## 5. 현재 시점의 판단
현재 프로젝트는 다음 단계에 도달했다고 본다.

- 369 / 눈치게임 기본 플레이 구조는 usable 상태
- 공통 UI 구조는 이미 재사용 가능한 수준
- 리액션 시스템은 단순 실험 단계가 아니라 **동기화 검증 완료 단계**
- 이제부터는 기능 추가보다 **리액션 UI polish + 하단 버튼 마감 + 다음 게임 준비** 순서가 맞다

즉, 지금은 구조를 흔드는 단계가 아니라
**리액션 연출과 UI 마감 품질을 높이는 단계**로 본다.

## 6. 다음 단계 우선순위

### 1순위(적용, 테스트 완료, 확인 필요)
**리액션 위치/겹침 마감**
- 같은 플레이어가 연속으로 눌렀을 때 덜 뭉치게
- 카드 위에 조금 더 떠 있는 느낌으로 조정
- 최대 3개 정도만 보이도록 제한
- 잘림 여부 확인

### 2순위
**369 하단 입력 버튼 시각 polish**
- 박수 / 만세 버튼을 이모지 중심으로 정리
- 버튼 텍스트/이모지 크기 및 높이 안정화
- 작은 화면에서도 하단 잘림이 없는지 확인

### 3순위
**리액션 연출 polish**
- 필요 시 fade + scale 미세 조정
- 리액션 종류별 시각 차별화 여부 검토
- 너무 과하지 않게 유지

### 4순위
**번데기 게임 착수 준비**
- 지금 당장은 바로 구현보다,
- 리액션/공통 UI 마감 이후에 구조적으로 들어가는 것이 맞음

## 7. 번데기 게임 착수 시점 판단
현재 판단은 아래와 같다.

- 지금 당장 번데기 게임 구현에 들어가기보다
- 리액션과 공통 UI 마감을 먼저 끝내는 것이 맞다

이유:
1. 공통 구조가 안정된 뒤 들어가야 반복 수정이 줄어든다
2. 번데기는 규칙/UI 복잡도가 369보다 더 높을 수 있다
3. 지금 리액션 시스템은 모든 게임이 공통으로 쓰게 될 기반 기능이다

즉,
- **지금 = 리액션/공통 UI 마감**
- **그 다음 = 번데기 게임 MVP 구조 착수**
